# Zotero 9 Compatibility Design

**GitHub Issue:** None

## Summary

`zotero-api-plus` is a TypeScript plugin that adds a small local HTTP API (`/api/plus`, `/api/plus/add-item-by-id`, `/api/plus/selected-collection`) on top of Zotero's built-in `Zotero.Server.LocalAPI` infrastructure. The migration broadens its manifest's `strict_max_version` from `8.*` to `9.*`, making the plugin installable in Zotero 9 without narrowing compatibility with Zotero 7 or 8. The same phase also refreshes three dev-only dependencies to versions aligned with the Zotero 9 release, removes one dead property (`permitBookmarklet`) that the LocalAPI router never reads, and corrects a MIME-type typo (`"plain/text"` → `"text/plain"`) discovered during design.

The second phase adds contract tests: three test files that instantiate each endpoint class directly and call `.run()`, covering the primary code paths and key error branches, plus one real-network smoke test that resolves a stable arXiv DOI through Zotero's translator pipeline and asserts a successful item add. These tests record the contract observed on Zotero 9, giving future Zotero version upgrades something concrete to break against. A third phase updates the README badge and replaces outdated `prefs.js` setup instructions with the current in-app toggle. The esbuild bundle target remains `firefox115` throughout to preserve runtime compatibility with Zotero 7's older engine. Compatibility with Zotero 7 and 8 is reasoned structurally rather than verified against installed binaries, which is explicitly out of scope.

## Definition of Done

The migration is complete when all of the following are true. Each entry is observable and falsifiable; the explicit exclusions bound the scope.

1. **Manifest accepts Zotero 9.** `addon/manifest.json` updates `strict_max_version` from `"8.*"` to `"9.*"`. `strict_min_version` remains `"6.999"` (preserves Z7/Z8 install eligibility). Falsifiable by `git diff`; the built `.xpi` installs on the local Z9 binary in normal (non-temporary) install mode without a "not compatible" rejection.

2. **Existing startup test still passes on Z9.** `npm run test` shows `1 passing` for `should have plugin instance defined` against the local Z9 flatpak binary at `/var/lib/flatpak/app/org.zotero.Zotero/.../files/share/zotero/zotero` using the dedicated dev profile at `/tmp/zotero-api-plus-dev-profile`.

3. **Three new in-process endpoint contract test files added and passing on Z9.** Each file instantiates the endpoint class directly and calls `.run()`. Multiple `it(...)` cases per file:
   - `Plus.run(...)` returns `[200, "text/plain", "Zotero Local API Plus is running."]` (after the item-8 typo fix).
   - `AddItemEndpoint.run({ data: {} })` returns `[400, "text/plain", "Error: No identifier provided"]` (no-identifier negative path, no network).
   - `AddItemEndpoint.run({ data: { identifier: "not-a-doi" } })` returns `[400, "text/plain", "Error: Could not parse identifier"]` (unparseable-identifier negative path, no network).
   - `GetSelectedCollectionEndpoint.run(...)` returns `[500, "text/plain", body]` when no collection is selected. Test asserts `status === 500`; body is checked tolerantly (matches either `"No Collection selected."` or `"Internal Server Error: ..."` in case `getActiveZoteroPane()` is null in the runner's state).
   Falsifiable: `npm run test` shows 5 passing total (existing `startup` test + 4 new `it` cases across 3 files).

4. **One real-network smoke test for `AddItemEndpoint` passes against a stable DOI.** Target: `10.48550/arXiv.1706.03762` ("Attention Is All You Need"; arXiv DOI; arXiv translator is the most reliable in Zotero core; paper is open-access). Test asserts `[200, "application/json", payload]` where `JSON.parse(payload).addedCount >= 1` and `titles.length >= 1`. Mocha per-test timeout extended to 30 000 ms (default 2 000 ms is too short for translator round-trip). **No skip-on-network-failure wrapper:** if the network is down or the translator regresses, the test fails — that is the correct signal. Falsifiable: `npm run test` shows 6 passing total when network is available.

5. **Three dev-deps refreshed; build/lint/typecheck stay green.** `package.json` updates:
   - `zotero-types`: `^4.1.0-beta.4` → `^4.1.2`
   - `zotero-plugin-toolkit`: `^5.1.0-beta.13` → `^5.1.2`
   - `zotero-plugin-scaffold`: `^0.8.2` → `^0.8.6`
   Falsifiable: `package.json` diff matches; `npm install && npm run lint:check && npm run build` all exit 0; `npx tsc --noEmit` reports `No errors found`.

6. **Dead `permitBookmarklet` line removed from `src/addon.ts`.** The line `permitBookmarklet = true;` on the `AddItemEndpoint` class is removed (the LocalAPI router does not consult this property — only the legacy connector server does, and we don't register there). Falsifiable: `git diff` shows the single-line removal; nothing else changes in that class; tests still pass.

7. **MIME-type typo fixed in `src/addon.ts:16`.** The `Plus` endpoint currently returns `"plain/text"` which is not a valid IANA MIME type (the correct value is `"text/plain"`). Single-line change: `"plain/text"` → `"text/plain"`. Falsifiable: `git diff src/addon.ts` shows exactly that one-line change; the new `endpoint-plus` test asserts `"text/plain"` and passes; `npm run test` exit 0.

8. **README.md updated for accuracy.** Two changes only:
   - Badge: the `Zotero-8` shield is replaced with `Zotero-7|8|9` (or equivalent wording covering the supported range).
   - "Usage" section: the outdated `prefs.js` paragraph (lines starting with "Ensure Zotero's local API is enabled…") is replaced with the current Z7+ UI toggle path: *Settings → Advanced → "Allow other applications on this computer to communicate with Zotero"*. Optionally include a `curl http://127.0.0.1:23119/api/plus` smoke command.
   Falsifiable: `git diff README.md` shows exactly those two changes; no other section is touched.

### Out of Scope (explicitly excluded)

These items are deliberately excluded from this design. They may become future work but will not be addressed in this migration.

- **Verification on Zotero 7 or Zotero 8.** Neither binary is installed locally; the design's compatibility claim for Z7/Z8 is reasoned (manifest range widened, dev-only deps don't ship in the `.xpi`, removed code was unused, build target unchanged) — not test-verified.
- **New endpoints, refactoring of existing endpoint logic, or any change to the Zotero translator pipeline.**
- **`esbuild target` change.** Stays at `firefox115` to preserve Z7's older engine compatibility.
- **Localization** (no `*.ftl` changes); **bootstrap.js** (scaffold-controlled); **CI workflows** (we are not pushing to GitHub for this migration); **version bump and release**; **any GitHub interaction**.
- **Translated READMEs** (`doc/README-zhCN.md`, `doc/README-frFR.md`) — only the English `README.md` is updated.

## Glossary

- **LocalAPI (`Zotero.Server.LocalAPI`)**: Zotero's built-in local HTTP server infrastructure, distinct from the legacy connector server. Endpoints registered here handle requests to `127.0.0.1:23119/api/...`.
- **`Zotero.Server.LocalAPI.Schema`**: The base class that all `zotero-api-plus` endpoint classes extend. Defines the `supportedMethods`, `supportedDataTypes`, and `run()` contract.
- **`strict_max_version` / `strict_min_version`**: Fields in `addon/manifest.json` that bound the Zotero version range in which the plugin will install. Unlike Firefox's soft `maxVersion`, `strict_max_version` causes Zotero to hard-reject the install if the running version exceeds the declared ceiling.
- **`.xpi`**: The ZIP-format package that Zotero consumes when installing a plugin. Produced by `npm run build` under `.scaffold/build/`.
- **`zotero-plugin-scaffold`**: Dev-only build and test orchestration tool. Launches Zotero against the dev profile, injects the plugin as a temporary add-on, runs mocha inside the Zotero process, and captures results. Controls `bootstrap.js` generation.
- **`zotero-plugin-toolkit`**: Dev-only utility library providing higher-level wrappers around Zotero's internal APIs (UI helpers, preference management, etc.). Does not ship in the `.xpi`.
- **`zotero-types`**: Dev-only TypeScript type definitions for Zotero's internal APIs. Used at compile time only; not bundled.
- **`typings/global.d.ts`**: Project-local TypeScript declaration file that augments `Zotero.Server.LocalAPI` to include the plugin's endpoint classes and re-declares `Server.LocalAPI.Schema` (absent from upstream `zotero-types`). The documented escape hatch for type errors surfaced by dep bumps.
- **Characterisation tests / Contract tests**: Both terms describe tests written to record existing behaviour rather than to specify desired behaviour, with the goal of surfacing regressions when a dependency or host application changes. This document uses **contract tests** for Phase 2 because the tests record the behaviour observed on Zotero 9 only — a true characterisation in the strict sense would require also having observed the same behaviour on Zotero 7 and 8, which is out of scope.
- **In-process contract tests**: Tests that call endpoint `.run()` directly within the same JavaScript process, bypassing HTTP transport. Faster and deterministic but do not cover server lifecycle, header parsing, or response framing.
- **Temporary add-on install**: A Zotero mechanism (used by scaffold during test runs) that loads a plugin without writing it to the profile's permanent extensions directory. It bypasses the `strict_max_version` check, which is why manual UAT with a normal install is required to verify the manifest change.
- **`esbuild target` (`firefox115`)**: The JavaScript engine version esbuild compiles the bundle down to. `firefox115` corresponds to Zotero 7's embedded Mozilla engine; keeping it here ensures the bundle is valid on Z7 even though Zotero 9 ships Mozilla 140 ESR.
- **`permitBookmarklet`**: A property on Zotero's legacy connector server endpoints that controls whether the connector's bookmarklet feature may invoke the endpoint. The LocalAPI router never reads this property; its presence on `AddItemEndpoint` was dead code inherited from an earlier template.
- **`Zotero.Translate.Search`**: Zotero's internal translator runner used by `AddItemEndpoint` to resolve a DOI (or other identifier) into bibliographic metadata via the appropriate translator.
- **tmpfs (`/tmp`)**: Linux in-memory filesystem. The dev profile directory (`/tmp/zotero-api-plus-dev-profile`) and data directory (`/tmp/zotero-api-plus-dev-data`) live here and are lost on reboot.
- **`prefs.js`**: A Firefox/Zotero profile file formerly used to enable the local HTTP server programmatically. Replaced in Zotero 7+ by an in-app toggle; the README's reference to it was stale.
- **Conventional Commits**: A commit message convention using prefixes such as `feat:`, `fix:`, `chore:`. Used by scaffold's automated release flow (`chore(publish): ...`) but not adopted for hand-written commits in this project.

## Architecture

A configuration/dependency change. The plugin's runtime architecture (three endpoint classes registered on `Zotero.Server.LocalAPI`, dispatched through `Zotero.Server.Endpoints`) is unchanged. The migration extends the manifest's accepted Zotero version range to include 9.x, refreshes three dev-only dependencies to versions current as of the Zotero 9 release date (2026-04-10), removes one line of dead code, fixes one MIME-type typo, and adds contract tests so future Zotero versions have something concrete to break against.

The plugin's API surface (`/api/plus`, `/api/plus/add-item-by-id`, `/api/plus/selected-collection`) is unchanged. The endpoint classes still extend `Zotero.Server.LocalAPI.Schema` (the only base class exposed via `typings/global.d.ts`), still return `[status, mime, body]` tuples, still use `Zotero.Translate.Search` for DOI resolution. No new files in `src/` or `addon/`; new files only in `test/` and one minor edit to `README.md`.

## Decision Record

### DR1: Grouped-by-concern sequencing (3 phases) over linear-by-DoD-item (7 phases)
**Status:** Accepted
**Confidence:** High
**Reevaluation triggers:** If a future migration touches more areas (UI, prefs, localisation), reconsider per-DoD-item granularity.

**Decision:** We chose three implementation phases (compatibility shift; characterization tests; documentation), each producing a single commit, over seven phases each producing their own commit.

**Consequences:**
- **Enables:** Three close-Zotero / `npm run test` cycles instead of seven; cleaner git history aligned with the work's natural concerns.
- **Prevents:** `git bisect` cannot pinpoint which individual DoD item caused a regression — the smallest revertible unit is "all of Phase 1" rather than a single line.

**Alternatives considered:**
- **Linear-by-DoD-item (7 phases):** Rejected because the workflow overhead (seven verification cycles) outweighs the bisect benefit for a migration where each item is individually low-risk.

### DR2: In-process contract tests + one real-network smoke test
**Status:** Accepted
**Confidence:** High
**Reevaluation triggers:** If endpoint behaviour starts depending on HTTP-layer concerns (CORS, Content-Length parsing, chunked encoding), the in-process tests will become insufficient.

**Decision:** We chose to instantiate endpoint classes directly and call `.run()` for contract verification, plus one real-network smoke test against the arXiv DOI `10.48550/arXiv.1706.03762` for end-to-end translator validation.

**Consequences:**
- **Enables:** Fast, deterministic contract tests (<50 ms each) covering negative cases without network dependency. One real translator run that catches Zotero-side regressions in the DOI resolution path.
- **Prevents:** No coverage of the HTTP transport layer (server lifecycle, header parsing, response framing). If a future Zotero update breaks `Zotero.Server`'s dispatch of `/api/plus/...` while the endpoint classes themselves remain correct, our tests will not catch it.

**Alternatives considered:**
- **Contract-only, no smoke test:** Rejected because a Zotero-side translator regression could break the production endpoint silently without any test signal.
- **Full HTTP integration:** Rejected because the additional scaffolding (enable `httpServer.enabled` in dev profile, manage server lifecycle in test setup/teardown) does not pay for itself given the existing temporary-install mechanism already exercises the registration path.

### DR3: Fix the MIME-type typo as part of this migration
**Status:** Accepted
**Confidence:** High
**Reevaluation triggers:** None — the typo is unambiguous; the IANA registration is `text/plain`.

**Decision:** We chose to fix `"plain/text"` → `"text/plain"` on `src/addon.ts:16` as DoD item 7, rather than encoding the typo as a characterization test or deferring it.

**Consequences:**
- **Enables:** The endpoint reports the correct MIME type to clients. No future migration carries the bug forward.
- **Prevents:** Nothing material. Most HTTP clients are lenient enough that the bug was invisible in practice.

**Risk acknowledged and dismissed:** in principle, fixing a long-standing bug can be a breaking change for any consumer that built logic around the bug. Here that risk is judged negligible because (a) `"plain/text"` is not a registered MIME type and is not produced by any widely-used server, so no compliant client would have hard-coded acceptance of it, and (b) `"text/plain"` is the IANA-registered value that any spec-following client already handles correctly. We are not aware of any consumer of `/api/plus` that depends on the malformed value; if one exists, the fix shifts them onto correct behaviour.

**Alternatives considered:**
- **Encode the bug as characterisation:** Rejected because it locks incorrect behaviour into the test suite and creates a future cleanup task.
- **Encode + add TODO comment:** Rejected for the same reason; the comment-then-fix-later cycle is friction with no benefit when the fix is a single line (typo).

### DR4: Preserve `firefox115` esbuild target
**Status:** Accepted
**Confidence:** High
**Reevaluation triggers:** If the supported-Zotero range is ever narrowed to drop Z7, bump target to the highest engine still supported.

**Decision:** We chose to leave `target: "firefox115"` in `zotero-plugin.config.ts:36` unchanged, even though Zotero 9 ships with Mozilla 140 ESR.

**Consequences:**
- **Enables:** Generated bundle remains compatible with Zotero 7's older Firefox 115 engine, preserving the "current support set" the user wants.
- **Prevents:** Use of newer JavaScript syntax/APIs (e.g. ECMAScript 2024+ features) that esbuild would only emit when targeting a newer engine.

**Alternatives considered:**
- **Bump to `firefox128` (Z8 engine) or `firefox140` (Z9 engine):** Rejected because either bump silently breaks Z7 install at runtime — not at install-check time.

### DR5: Smoke test fails on network unavailability
**Status:** Accepted
**Confidence:** High
**Reevaluation triggers:** If this design is later adopted in unattended CI without network access, revisit (skip-on-failure may then become preferable).

**Decision:** We chose to let the arXiv smoke test fail (not skip) when the network is unreachable or the translator returns no items.

**Consequences:**
- **Enables:** Honest test signal — a real translator regression cannot hide as "always skipped". `npm run test` failure means something the user cares about is broken.
- **Prevents:** Running `npm run test` cleanly while offline. Mitigated by `npm run test -- --grep "^((?!smoke).)*$"` for offline development.

**Alternatives considered:**
- **Skip-on-network-failure with named console message:** Rejected because intermittent network flakiness could mask a true translator regression as "always skipped".

## Existing Patterns

Investigation found a small but consistent set of patterns in `src/addon.ts` and `test/startup.test.ts`. This design follows them.

**Test framework.** `test/startup.test.ts` uses mocha `describe`/`it` with chai's `assert` style: `import { assert } from "chai"; import { config } from "../package.json";`. New tests follow exactly this import pattern and structure.

**Endpoint contract.** Existing endpoints (`Plus`, `AddItemEndpoint`, `GetSelectedCollectionEndpoint` in `src/addon.ts`) all extend `Zotero.Server.LocalAPI.Schema`, declare `supportedMethods` (and `supportedDataTypes` for POST), and implement `async run(...)` returning a `[status: number, mime: string, body: string]` tuple. New tests verify against this exact contract — no schema changes, no Response objects.

**Type augmentation escape hatch.** `typings/global.d.ts:20-37` augments `Zotero.Server.LocalAPI` with the project's endpoint classes. If `zotero-types@4.1.2` introduces conflicting type definitions, the project's existing pattern is to resolve via this file (the same way `Server.LocalAPI.Schema` is declared there today).

**Comments.** Bilingual (English + Chinese) comments inherited from upstream. New additions are English-only; existing Chinese comments are preserved unchanged.

**Commit style.** `git log --oneline -10` shows mixed Conventional Commits (`chore(publish): ...` from scaffold's release flow) and plain imperative (`add GetSelectedCollectionEndpoint`, `update readme`). New commits in this migration use plain imperative — they are not release commits.

**Linting.** `eslint.config.mjs` extends `@zotero-plugin/eslint-config` with one local override (`@typescript-eslint/no-unused-vars` disabled for `**/*.ts` because the upstream template ships unused examples). Prettier is configured in `package.json` (`printWidth: 80`, `tabWidth: 2`, `endOfLine: "lf"`). All new files comply with both.

## Additional Considerations

**Zotero must be closed during each test cycle.** Scaffold launches Zotero against the dev profile at `/tmp/zotero-api-plus-dev-profile` to install the plugin and run mocha. A running user-session Zotero will conflict over the SQLite lock. Each phase that runs `npm run test` (Phases 1 and 2) requires the user to quit Zotero first.

**Dev profile and data directories live in tmpfs.** `/tmp/zotero-api-plus-dev-profile` and `/tmp/zotero-api-plus-dev-data` survive only until the next reboot. The `.env` file at the project root references both paths. Recreate after reboot with `mkdir -p /tmp/zotero-api-plus-dev-profile /tmp/zotero-api-plus-dev-data`.

**Manual final verification (operator-performed, not automated).** After all three phases commit, build a real `.xpi` (`npm run build` produces it under `.scaffold/build/`), install it into a *normal* (non-temporary) Z9 profile via Tools → Plugins, confirm Zotero accepts the install (the `strict_max_version` bump is what this checks — temporary install bypasses that check, so the test suite cannot verify it), then `curl http://127.0.0.1:23119/api/plus` and confirm the response body is `Zotero Local API Plus is running.` with `Content-Type: text/plain`.

**Zotero 7 / Zotero 8 compatibility is reasoned, not tested.** Neither binary is installed locally. The compatibility claim rests on these structural arguments, **each of which is an assumption rather than an empirical observation against Z7 or Z8 binaries**: (i) the manifest range is widened, not narrowed — assumes Z7/Z8 honour `strict_max_version` per the Mozilla WebExtension manifest spec that Zotero's add-on manager inherits via Firefox's `nsIAddonManager`; (ii) the dev-deps do not ship in the `.xpi` — verifiable from `zotero-plugin.config.ts` (`assets: ["addon/**/*.*"]` is the only thing packaged); (iii) the removed `permitBookmarklet` property was never read by the LocalAPI router — verified across the `zotero/zotero` source on `main`, but not pinned to the Z7 or Z8 release branches; (iv) the MIME typo fix corrects a genuine bug rather than introducing one — see DR3; (v) the esbuild target is unchanged. If a future user reports Z7 or Z8 breakage, that is in-scope as a bug to triage but out-of-scope for this design's verification gate.

**The smoke test is operable only in developer-local environments with live network.** A CI runner that lacks outbound network access, or that lacks an installed Zotero binary at all, cannot pass the smoke test (`z9-compat.AC4`). If CI is ever revived, the smoke test must be either filtered via `--grep` or the runner must be provisioned with both network access and a Zotero install. Independently of this design, the existing `.github/workflows/ci.yml` `test` job is non-functional today — `zotero-plugin-dev/workflows/setup-js@main` does not install Zotero — and would fail before reaching any of our new tests; fixing that is upstream work and out of scope for this migration.
