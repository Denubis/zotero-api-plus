# Zotero 9 Compatibility Design

**GitHub Issue:** None

## Summary

`zotero-api-plus` is a TypeScript plugin that adds a small local HTTP API (`/api/plus`, `/api/plus/add-item-by-id`, `/api/plus/selected-collection`) on top of Zotero's built-in `Zotero.Server.LocalAPI` infrastructure. The migration broadens its manifest's `strict_max_version` from `8.*` to `9.*`, making the plugin installable in Zotero 9 without narrowing compatibility with Zotero 7 or 8. The same phase also refreshes three dev-only dependencies to versions aligned with the Zotero 9 release ŌĆö each bumped **one at a time, after reading its changelog**, with its own verification cycle and commit (see DR1), because two of the three cross a known semver carve-out (`zotero-plugin-scaffold` is pre-1.0; `zotero-plugin-toolkit` is a betaŌåÆstable transition). The phase finishes by removing one dead property (`permitBookmarklet`) that the LocalAPI router never reads, and correcting a MIME-type typo (`"plain/text"` ŌåÆ `"text/plain"`) discovered during design.

The second phase adds tests at three levels. A **preflight probe** converts two load-bearing test-harness assumptions (the scaffold `waitForPlugin` gate has populated the endpoint namespace by `it`-time; `this.timeout()` propagates to `it` callbacks) from background facts into positive controls, so a harness failure surfaces as a clear message rather than an opaque `TypeError`. **In-process contract tests** instantiate each endpoint class directly and call `.run()`, covering the primary code paths and key error branches, plus one real-network smoke test that resolves a stable arXiv DOI through Zotero's translator pipeline. **One HTTP dispatch test** (see DR2, revised) drives `Zotero.Server`'s real request dispatch over `127.0.0.1:23124` (scaffold's test-profile local-API port) ŌĆö a `GET /api/plus` and a `POST /api/plus/add-item-by-id` with an empty body ŌĆö to cover the routing surface that the in-process tests bypass, including the path where a removed property like `permitBookmarklet` would be read. The HTTP test is deliberately network-free (the empty-body POST returns 400 before any translator call), so the only network dependency stays isolated in the one smoke test. These tests record the contract observed on Zotero 9, giving future Zotero version upgrades something concrete to break against. A third phase updates the README badge and replaces outdated `prefs.js` setup instructions with the current in-app toggle. The esbuild bundle target remains `firefox115` throughout to preserve runtime compatibility with Zotero 7's older engine. Compatibility with Zotero 7 and 8 is reasoned structurally rather than verified against installed binaries, which is explicitly out of scope.

### Out of Scope (explicitly excluded)

These items are deliberately excluded from this design. They may become future work but will not be addressed in this migration.

- **Verification on Zotero 7 or Zotero 8.** Neither binary is installed locally; the design's compatibility claim for Z7/Z8 is reasoned (manifest range widened, dev-only deps don't ship in the `.xpi`, removed code was unused, build target unchanged) ŌĆö not test-verified.
- **New endpoints, refactoring of existing endpoint logic, or any change to the Zotero translator pipeline.**
- **`esbuild target` change.** Stays at `firefox115` to preserve Z7's older engine compatibility.
- **Localization** (no `*.ftl` changes); **bootstrap.js** (scaffold-controlled).
- **CI workflow changes.** The existing `.github/workflows/ci.yml` `test` job is non-functional today (it does not install a Zotero binary) and stays untouched. The PR (see below) will therefore show a red `test` check; that is expected and not a blocker for self-merge.
- **Version bump and release.** Not required for the PR. A tagged `0.3.0` release via `npm run release` is a separate, optional follow-up.
- **Upstream contribution.** No PR to `GOKORURI007/zotero-api-plus`. This migration targets the fork only.
- **Translated READMEs** (`doc/README-zhCN.md`, `doc/README-frFR.md`) ŌĆö only the English `README.md` is updated.

**In scope (revised ŌĆö branch fate, per critical review F2):** after the three phases commit and the operator UAT (AC1.3) passes, the `z9-compat` branch is pushed to `origin` and a pull request is opened against the fork's own `main` (`Denubis/zotero-api-plus`). The operator self-merges. This is the defined post-execution path the original design left unstated.

## Glossary

- **LocalAPI (`Zotero.Server.LocalAPI`)**: Zotero's built-in local HTTP server infrastructure, distinct from the legacy connector server. Endpoints registered here handle requests to `127.0.0.1:23119/api/...`.
- **`Zotero.Server.LocalAPI.Schema`**: The base class that all `zotero-api-plus` endpoint classes extend. Defines the `supportedMethods`, `supportedDataTypes`, and `run()` contract.
- **`strict_max_version` / `strict_min_version`**: Fields in `addon/manifest.json` that bound the Zotero version range in which the plugin will install. Unlike Firefox's soft `maxVersion`, `strict_max_version` causes Zotero to hard-reject the install if the running version exceeds the declared ceiling.
- **`.xpi`**: The ZIP-format package that Zotero consumes when installing a plugin. Produced by `npm run build` under `.scaffold/build/`.
- **`zotero-plugin-scaffold`**: Dev-only build and test orchestration tool. Launches Zotero against the dev profile, injects the plugin as a temporary add-on, runs mocha inside the Zotero process, and captures results. Controls `bootstrap.js` generation.
- **`zotero-plugin-toolkit`**: Dev-only utility library providing higher-level wrappers around Zotero's internal APIs (UI helpers, preference management, etc.). Does not ship in the `.xpi`.
- **`zotero-types`**: Dev-only TypeScript type definitions for Zotero's internal APIs. Used at compile time only; not bundled.
- **`typings/global.d.ts`**: Project-local TypeScript declaration file that augments `Zotero.Server.LocalAPI` to include the plugin's endpoint classes and re-declares `Server.LocalAPI.Schema` (absent from upstream `zotero-types`). The documented escape hatch for type errors surfaced by dep bumps.
- **Characterisation tests / Contract tests**: Both terms describe tests written to record existing behaviour rather than to specify desired behaviour, with the goal of surfacing regressions when a dependency or host application changes. This document uses **contract tests** for Phase 2 because the tests record the behaviour observed on Zotero 9 only ŌĆö a true characterisation in the strict sense would require also having observed the same behaviour on Zotero 7 and 8, which is out of scope.
- **In-process contract tests**: Tests that call endpoint `.run()` directly within the same JavaScript process, bypassing HTTP transport. Faster and deterministic but do not cover server lifecycle, header parsing, response framing, or `Zotero.Server`'s dispatch/routing.
- **HTTP dispatch test**: A test that makes a real HTTP request to `127.0.0.1:23124/api/plus/...` (scaffold test-profile port) so that `Zotero.Server`'s request routing (URL match, method/`supportedDataTypes` enforcement, response framing) is exercised ŌĆö the surface in-process `.run()` calls bypass. Introduced in the revised DR2; kept network-free.
- **Preflight probe**: A small `describe`/`it` block run first that asserts the test harness's load-bearing assumptions (endpoint namespace populated; `this.timeout()` propagation) so a harness fault produces a clear message instead of an opaque `TypeError` in a later contract test.
- **`extensions.zotero.httpServer.localAPI.enabled` / `...httpServer.port`**: Zotero preferences that turn the local API server on and set its port. Scaffold's test profile sets these to `true` and `23124` automatically (confirmed in Phase 1), so the HTTP dispatch test reaches `127.0.0.1:23124` with no enablement code. A normal Zotero install uses the default port 23119.
- **Temporary add-on install**: A Zotero mechanism (used by scaffold during test runs) that loads a plugin without writing it to the profile's permanent extensions directory. It bypasses the `strict_max_version` check, which is why manual UAT with a normal install is required to verify the manifest change.
- **`esbuild target` (`firefox115`)**: The JavaScript engine version esbuild compiles the bundle down to. `firefox115` corresponds to Zotero 7's embedded Mozilla engine; keeping it here ensures the bundle is valid on Z7 even though Zotero 9 ships Mozilla 140 ESR.
- **`permitBookmarklet`**: A property on Zotero's legacy connector server endpoints that controls whether the connector's bookmarklet feature may invoke the endpoint. The LocalAPI router never reads this property; its presence on `AddItemEndpoint` was dead code inherited from an earlier template.
- **`Zotero.Translate.Search`**: Zotero's internal translator runner used by `AddItemEndpoint` to resolve a DOI (or other identifier) into bibliographic metadata via the appropriate translator.
- **tmpfs (`/tmp`)**: Linux in-memory filesystem. The dev profile directory (`/tmp/zotero-api-plus-dev-profile`) and data directory (`/tmp/zotero-api-plus-dev-data`) live here and are lost on reboot.
- **`prefs.js`**: A Firefox/Zotero profile file formerly used to enable the local HTTP server programmatically. Replaced in Zotero 7+ by an in-app toggle; the README's reference to it was stale.
- **Conventional Commits**: A commit message convention using prefixes such as `feat:`, `fix:`, `chore:`. Used by scaffold's automated release flow (`chore(publish): ...`) but not adopted for hand-written commits in this project.

## Architecture

A configuration/dependency change. The plugin's runtime architecture (three endpoint classes registered on `Zotero.Server.LocalAPI`, dispatched through `Zotero.Server.Endpoints` ŌĆö see `src/addon.ts` `registerEndpoints()`) is unchanged. The migration extends the manifest's accepted Zotero version range to include 9.x, refreshes three dev-only dependencies to versions current as of the Zotero 9 release date (2026-04-10), removes one line of dead code, fixes one MIME-type typo, and adds tests (preflight harness controls, in-process contract tests, one HTTP dispatch test against the real `Zotero.Server.Endpoints` routing, and one network smoke test) so future Zotero versions have something concrete to break against.

The plugin's API surface (`/api/plus`, `/api/plus/add-item-by-id`, `/api/plus/selected-collection`) is unchanged. The endpoint classes still extend `Zotero.Server.LocalAPI.Schema` (the only base class exposed via `typings/global.d.ts`), still return `[status, mime, body]` tuples, still use `Zotero.Translate.Search` for DOI resolution. No new files in `src/` or `addon/`; new files only in `test/` and one minor edit to `README.md`.

## Decision Record

### DR1: Three phases by concern, but Phase 1 bumps dependencies one at a time (revised)

**Status:** Accepted ŌĆö **revised** after critical review m3 (dep bumps are not the uniformly-low-risk changes the original framing assumed).
**Confidence:** High
**Reevaluation triggers:** If a future migration's deps are all stable post-1.0 releases with clean changelogs, the one-at-a-time discipline can relax back to a single combined bump commit.

**Decision:** Three implementation phases (compatibility shift; tests; documentation). Phases 2 and 3 each produce a single commit. **Phase 1 is no longer a single commit:** the three dependency bumps are applied one at a time ŌĆö read changelog ŌåÆ bump one package ŌåÆ `npm install` ŌåÆ full verification ŌåÆ commit ŌĆö in ascending risk order (`zotero-types`, then `zotero-plugin-toolkit`, then `zotero-plugin-scaffold`), followed by a fourth commit for the manifest widening + dead-code removal + MIME fix. Phase 1 therefore yields **four commits**.

**Why the revision:** the original "low risk, one commit" framing did not survive scrutiny. `zotero-plugin-scaffold` is pre-1.0 (semver gives no API-stability guarantee in `0.x`) and controls the test runner and dev-profile launch; `zotero-plugin-toolkit` crosses a betaŌåÆstable boundary where features are sometimes cut. Bundling all three into one commit means a runner regression cannot be attributed to a single package without manual bisection inside the commit. The HTTP dispatch test (DR2) is especially sensitive to the scaffold bump, which is why scaffold is bumped last and alone.

**Consequences:**

- **Enables:** `git bisect` (and plain `git revert`) can isolate a regression to one dependency; the changelog-review step is forced rather than optional; the scaffold bump ŌĆö the riskiest ŌĆö is the last variable changed before Phase 2's HTTP test depends on it.
- **Prevents:** Phase 1 now runs four verification cycles instead of one (each requires Zotero closed). The DR1.2 commit-count check becomes content-based (which messages exist), not a hardcoded number (per critical review m4).

**Alternatives considered:**

- **Single combined Phase 1 commit (the original plan):** Rejected on review ŌĆö see "Why the revision."
- **Linear-by-DoD-item (7 phases):** Still rejected ŌĆö the documentation and dead-code items remain genuinely low-risk and do not each need their own verification cycle.

### DR2: In-process contract tests + one HTTP dispatch test + one real-network smoke test (revised)

**Status:** Accepted ŌĆö **revised** after critical review I1/I3/I5 (the in-process test surface does not overlap with the changes being made).
**Confidence:** High (Phase 1 execution confirmed scaffold's test profile already runs the local API on port 23124 ŌĆö the feasibility precondition is satisfied, no enablement code needed).
**Reevaluation triggers:** If a future scaffold version stops enabling the local API in the test profile, or the HTTP server proves unreachable in-harness, the HTTP test is re-scoped and this DR reverts toward its original in-process-only form (documented, not silent).

**Decision:** Three test levels: (1) in-process contract tests that instantiate endpoint classes and call `.run()`; (2) **one HTTP dispatch test** that drives `Zotero.Server`'s real request routing over `127.0.0.1:23124` (scaffold test-profile port) with two network-free cases (`GET /api/plus`; empty-body `POST /api/plus/add-item-by-id` ŌåÆ 400); (3) one real-network smoke test against the arXiv DOI `10.48550/arXiv.1706.03762`. A preflight probe (DoD item 10) guards the harness assumptions the in-process tests rely on.

**Why the revision:** the original DR2 rejected HTTP integration as "not paying for itself given the temporary-install mechanism already exercises the registration path." Critical review showed that reasoning is too weak for _this_ migration: the changes being made (removing a property the dispatch layer could read; bumping the scaffold that launches the runner) live precisely on the `Zotero.Server` dispatch surface, which `.run()` calls bypass. "The registration path runs" is not the same as "a request dispatches correctly." A single HTTP test buys coverage of the one surface the migration most plausibly disturbs. We keep it network-free so it adds coverage without adding flakiness.

**Consequences:**

- **Enables:** Automated coverage of the `Zotero.Server` dispatch path ŌĆö GET and POST routing, `supportedDataTypes` enforcement, and the property-read surface relevant to the `permitBookmarklet` removal. A dispatch regression now fails a test instead of waiting for production.
- **Prevents / costs:** Requires the HTTP server enabled in the dev profile (a precondition the original design deliberately avoided). Adds harness setup and a spike (DoD item 9). Still does not cover the connector/bookmarklet path (the plugin never registers there) ŌĆö so it is a strict improvement, not total proof the removed property is unread.

**Alternatives considered:**

- **In-process only (the original DR2):** Rejected on review ŌĆö the test surface does not discriminate against the failure modes the changes introduce.
- **Full HTTP integration for every endpoint:** Rejected ŌĆö one GET + one POST through dispatch is enough to characterise the routing surface; replicating every contract case over HTTP duplicates the in-process tests for little marginal signal.

### DR3: Fix the MIME-type typo as part of this migration

**Status:** Accepted
**Confidence:** High
**Reevaluation triggers:** None ŌĆö the typo is unambiguous; the IANA registration is `text/plain`.

**Decision:** We chose to fix `"plain/text"` ŌåÆ `"text/plain"` on `src/addon.ts:16` as DoD item 7, rather than encoding the typo as a characterization test or deferring it.

**Consequences:**

- **Enables:** The endpoint reports the correct MIME type to clients. No future migration carries the bug forward.
- **Prevents:** Nothing material. Most HTTP clients are lenient enough that the bug was invisible in practice.

**Risk acknowledged and dismissed:** in principle, fixing a long-standing bug can be a breaking change for any consumer that built logic around the bug. Here that risk is judged negligible because (a) `"plain/text"` is not a registered MIME type and is not produced by any widely-used server, so no compliant client would have hard-coded acceptance of it, and (b) `"text/plain"` is the IANA-registered value that any spec-following client already handles correctly. We are not aware of any consumer of `/api/plus` that depends on the malformed value; if one exists, the fix shifts them onto correct behaviour.

**Alternatives considered:**

- **Encode the bug as characterisation:** Rejected because it locks incorrect behaviour into the test suite and creates a future cleanup task.
- **Encode + add TODO comment:** Rejected for the same reason; the comment-then-fix-later cycle is friction with no benefit when the fix is a single line (typo).

### DR4: Preserve `firefox115` esbuild target

**Status:** Accepted
**Confidence:** High
**Reevaluation triggers:** If the supported-Zotero range is ever narrowed to drop Z7, bump target to the highest engine still supported.

**Decision:** We chose to leave `target: "firefox115"` in `zotero-plugin.config.ts:36` unchanged, even though Zotero 9 ships with Mozilla 140 ESR.

**Consequences:**

- **Enables:** Generated bundle remains compatible with Zotero 7's older Firefox 115 engine, preserving the "current support set" the user wants.
- **Prevents:** Use of newer JavaScript syntax/APIs (e.g. ECMAScript 2024+ features) that esbuild would only emit when targeting a newer engine.

**Alternatives considered:**

- **Bump to `firefox128` (Z8 engine) or `firefox140` (Z9 engine):** Rejected because either bump silently breaks Z7 install at runtime ŌĆö not at install-check time.

### DR5: Smoke test fails on network unavailability

**Status:** Accepted
**Confidence:** High
**Reevaluation triggers:** If this design is later adopted in unattended CI without network access, revisit (skip-on-failure may then become preferable).

**Decision:** We chose to let the arXiv smoke test fail (not skip) when the network is unreachable or the translator returns no items.

**Consequences:**

- **Enables:** Honest test signal ŌĆö a real translator regression cannot hide as "always skipped". `npm run test` failure means something the user cares about is broken.
- **Prevents:** Running `npm run test` cleanly while offline. Mitigated by `npm run test -- --grep "^((?!smoke).)*$"` for offline development.

**Alternatives considered:**

- **Skip-on-network-failure with named console message:** Rejected because intermittent network flakiness could mask a true translator regression as "always skipped".

**Triage runbook (added per critical review I2/F3).** A smoke `Ō£¢` is necessary but not sufficient to conclude "translator regression." Five causes produce the same failure: (1) arXiv translator regression on Z9; (2) network/DNS outage; (3) the DOI withdrawn or its metadata reshaped; (4) arXiv rate-limited the runner; (5) a scaffold/mocha harness fault from the dep bumps. The operator discriminates before acting:

1. **Network vs. everything else:** `curl -I https://arxiv.org/abs/1706.03762` ŌĆö non-200/timeout points at (2) or (4); rerun after a pause.
2. **Harness vs. endpoint:** confirm the preflight probe and the in-process + HTTP contract tests still pass (`npm run test` with the smoke `it` filtered out). If those fail too, suspect (5) ŌĆö the scaffold bump ŌĆö not the translator.
3. **Translator vs. DOI:** if network is up and the harness is healthy, the residual is (1) or (3). Check whether the arXiv translator changed (zotero/translators) and whether the DOI still resolves in a browser.

**Operator options after triage (per F3):** (a) rerun if transient (2)/(4); (b) swap `TEST_DOI` to another stable open-access DOI if (3); (c) hold the `z9-compat` branch as not-mergeable and file an upstream translator issue if a genuine (1). The decision is the operator's; the test does not auto-skip or auto-retry.

## Existing Patterns

Investigation found a small but consistent set of patterns in `src/addon.ts` and `test/startup.test.ts`. This design follows them.

**Test framework.** `test/startup.test.ts` uses mocha `describe`/`it` with chai's `assert` style: `import { assert } from "chai"; import { config } from "../package.json";`. New tests follow exactly this import pattern and structure.

**Endpoint contract.** Existing endpoints (`Plus`, `AddItemEndpoint`, `GetSelectedCollectionEndpoint` in `src/addon.ts`) all extend `Zotero.Server.LocalAPI.Schema`, declare `supportedMethods` (and `supportedDataTypes` for POST), and implement `async run(...)` returning a `[status: number, mime: string, body: string]` tuple. New tests verify against this exact contract ŌĆö no schema changes, no Response objects.

**Type augmentation escape hatch.** `typings/global.d.ts:20-37` augments `Zotero.Server.LocalAPI` with the project's endpoint classes. If `zotero-types@4.1.2` introduces conflicting type definitions, the project's existing pattern is to resolve via this file (the same way `Server.LocalAPI.Schema` is declared there today).

**Comments.** Bilingual (English + Chinese) comments inherited from upstream. New additions are English-only; existing Chinese comments are preserved unchanged.

**Commit style.** `git log --oneline -10` shows mixed Conventional Commits (`chore(publish): ...` from scaffold's release flow) and plain imperative (`add GetSelectedCollectionEndpoint`, `update readme`). New commits in this migration use plain imperative ŌĆö they are not release commits.

**Linting.** `eslint.config.mjs` extends `@zotero-plugin/eslint-config` with one local override (`@typescript-eslint/no-unused-vars` disabled for `**/*.ts` because the upstream template ships unused examples). Prettier is configured in `package.json` (`printWidth: 80`, `tabWidth: 2`, `endOfLine: "lf"`). All new files comply with both.

## Additional Considerations

**Zotero does NOT need to be closed during test cycles (corrected during Phase 1 execution).** Scaffold launches an isolated `no-remote` Zotero against its own profile (`.scaffold/test/profile`, data dir `.scaffold/test/data`) on local-API port **23124** ŌĆö independent of a normal session (port 23119), so the two coexist. (The earlier claim that they conflict over the SQLite lock, and the `/tmp/...` profile path from `.env`, do not apply to the `npm run test` path.) **Test-run method:** scaffold does NOT self-exit after tests finish (even 0.8.6), so run `npm run test` under `timeout`, treat `npm` exit `124` as expected, and read success from the log line `Test run completed - N passed` (with `0 failing`); never pipe through `tail` (it flushes only at EOF, which never comes). Clean up any straggler test instance with `for pid in $(pgrep -a zotero | grep scaffold/test | awk '{print $1}'); do kill "$pid"; done` (kills only `.scaffold/test` instances, never a normal session).

**Dev profile and data directories live in tmpfs.** `/tmp/zotero-api-plus-dev-profile` and `/tmp/zotero-api-plus-dev-data` survive only until the next reboot. The `.env` file at the project root references both paths. Recreate after reboot with `mkdir -p /tmp/zotero-api-plus-dev-profile /tmp/zotero-api-plus-dev-data`.

**Manual final verification (operator-performed, not automated).** **Operator: Brian (repository owner). Trigger: after the Phase 3 commit, before pushing `z9-compat` / opening the PR. Recording: append `YYYY-MM-DD: PASS/FAIL ŌĆö initials` to `uat-requirements.md`.** (F1: the handoff is now owned and time-anchored.) Steps: build a real `.xpi` (`npm run build` produces it under `.scaffold/build/`), install it into a _normal_ (non-temporary) Z9 profile via Tools ŌåÆ Plugins, confirm Zotero accepts the install (the `strict_max_version` bump is what this checks ŌĆö temporary install bypasses that check, so the test suite cannot verify it), then `curl http://127.0.0.1:23119/api/plus` and confirm the body is `Zotero Local API Plus is running.` with `Content-Type: text/plain`. Also `curl -X POST -H 'Content-Type: application/json' -d '{"identifier":"10.48550/arXiv.1706.03762"}' http://127.0.0.1:23119/api/plus/add-item-by-id` against the real profile ŌĆö this is the one place the connector/bookmarklet-adjacent POST path is exercised on a normal install, complementing the automated HTTP dispatch test (AC9) which runs against the dev profile.

**Zotero 7 / Zotero 8 compatibility is reasoned, not tested.** Neither binary is installed locally. The compatibility claim rests on these structural arguments, **each of which is an assumption rather than an empirical observation against Z7 or Z8 binaries**: (i) the manifest range is widened, not narrowed ŌĆö assumes Z7/Z8 honour `strict_max_version` per the Mozilla WebExtension manifest spec that Zotero's add-on manager inherits via Firefox's `nsIAddonManager`; (ii) the dev-deps do not ship in the `.xpi` ŌĆö verifiable from `zotero-plugin.config.ts` (`assets: ["addon/**/*.*"]` is the only thing packaged); (iii) the removed `permitBookmarklet` property was never read by the LocalAPI router. **Per critical review I1, this claim must be pinned to a Z9 source tag, not `main`:** the Phase 1 changelog/source step records the exact `zotero/zotero` tag (e.g. the Z9 release tag) and the file:line in `Zotero.Server.LocalAPI`'s dispatch confirming the property is unread; if that citation cannot be made, the claim is hedged to "reasoned, not source-verified." Note also that no test (in-process or HTTP) exercises the connector/bookmarklet path ŌĆö the HTTP dispatch test (AC9) covers only the LocalAPI `/api/...` dispatch. Z7/Z8 remain un-pinned (out of scope); (iv) the MIME typo fix corrects a genuine bug rather than introducing one ŌĆö see DR3; (v) the esbuild target is unchanged. If a future user reports Z7 or Z8 breakage, that is in-scope as a bug to triage but out-of-scope for this design's verification gate.

**The smoke test is operable only in developer-local environments with live network.** A CI runner that lacks outbound network access, or that lacks an installed Zotero binary at all, cannot pass the smoke test (`z9-compat.AC4`). If CI is ever revived, the smoke test must be either filtered via `--grep` or the runner must be provisioned with both network access and a Zotero install. Independently of this design, the existing `.github/workflows/ci.yml` `test` job is non-functional today ŌĆö `zotero-plugin-dev/workflows/setup-js@main` does not install Zotero ŌĆö and would fail before reaching any of our new tests; fixing that is upstream work and out of scope for this migration.
